# 方法论：为本主题动态搭建争点体系

> 证券虚假陈述专题有固定的"侵权构成要件五维 14 问"作脊柱；通用类案研究的主题千变万化，**没有现成脊柱可抄**。因此通用版把"搭争点体系"做成**每次研究检查点 1 的一道必经工序**：先据本主题构造一套问题清单与倾向标签，作为后续编码、统计、报告第二章的共同骨架。产出文件 `01b_issues.md`，在检查点 1 连同要素与关键词一并交律师审核。
>
> 与 `keyword-strategy.md`（怎么把案件搜进来）、`relevance-screening.md`（怎么把无关案件筛出去）并列：本文件解决"搜进来之后，沿哪些问题去读、去编码、去成文"。

## 一、为什么先搭体系，再检索编码

- **统计需要离散键**：`run_analytics.py` 的维度五（焦点立场分布）、争点频次图都按"焦点名"聚合。焦点名若每条案件随手另起，统计就碎成一地，出不了图、汇不成表。
- **报告需要脊柱**：报告第二章按"争议焦点"逐节四段展开（见报告模板）。焦点体系即章节骨架，先定好，写作才不散。
- **编码需要一字不差的键名**：编码 `05_enriched_cases.json` 的 `焦点立场`/`抗辩` 用的焦点名、抗辩名，必须与本文件清单**逐字一致**，否则脚本聚合不到。

## 二、争点体系的三个层级

一套完整的争点体系含三类离散标签，编码时分别落到不同字段：

1. **争议焦点（issues）** —— 本类案件反复争夺的法律问题，3–6 个为宜。每个焦点是报告第二章的一节、维度五统计的一个聚合键。
   - 落到编码字段：`焦点立场: {"<焦点名>": {"立场": "...", "理由": "..."}}`。
2. **倾向标签（每个焦点的立场取值集）** —— 该焦点下法院可能站的几种立场，离散、互斥、可穷举。
   - 通用三档兜底：`支持原告 / 支持被告 / 未评述`；多数主题可在此之上细化（见下例）。
3. **抗辩理由（defenses）** —— 被告方反复提出的抗辩，供维度二"抗辩有效性"统计成功率。
   - 落到编码字段：`抗辩: [{"理由": "<抗辩名>", "是否被采纳": true/false}]`。

## 三、从案情要素抽取争点的套路

在 `01_elements.md`（要素）与预研检索结果的基础上，按下列顺序逼出焦点：

1. **看权利义务的交锋点**：原告主张什么权利/责任，被告凭什么对抗，二者正面相撞处即焦点（如"平台是否尽到合理注意义务")。
2. **看法律要件的争夺**：把案由对应的请求权基础拆成构成要件，逐件问"这一要件本类案件常不常争"。常争的要件升格为焦点（如侵权之诉的"过错认定""因果关系""损害范围"）。
3. **看抗辩的反复出现**：被告常打的几张牌，既是 defenses，往往也反向定义一个焦点（"技术中立"抗辩 ↔ "平台过错认定"焦点）。
4. **看裁量空间**：金额、比例、期限等法院有自由裁量的地方，常自成一个焦点（"判赔数额的认定")。

抽到的候选焦点用一句话写清：**争什么、为何重要、两端立场各是什么**。然后收敛到 3–6 个——太多则报告碎、统计稀，太少则深度不够。

## 四、倾向标签的设计规则

- **离散、互斥、可穷举**：一个焦点下的标签彼此不重叠，合起来覆盖绝大多数案件；确需新值时保持口径一致，不要近义词并存（"构成侵权"和"侵权成立"必须二选一固定）。
- **够细以承载差异，够粗以汇成统计**：标签太粗（只有支持/不支持）看不出分歧；太细（每案一个）统计不动。以"能让维度五的分布图说话"为准。
- **数值另存**：金额、比例（如系统风险扣除比例、过错责任比例）这类连续量不要塞进标签，单列数值字段（仿证券版 `系统风险扣除比例`），便于求分布、中位数。

## 五、示例（算法推荐短视频侵权，仅作样式参考，勿照搬到别的主题）

| 争议焦点 | 倾向标签取值集 |
|---|---|
| 平台合理注意义务认定 | 课以较高注意义务 / 一般注意义务 / 仅通知删除义务 |
| 算法推荐与过错的关系 | 算法推荐加重过错 / 算法推荐不影响过错 / 未评述 |
| 通知—必要措施是否到位 | 措施不足构成过错 / 措施充分免责 / 通知不合格 |
| 判赔数额的认定 | （数值另存 `判赔金额`）+ 标签：高于法定区间 / 法定区间内 / 酌减 |

对应的 **抗辩理由清单（defenses）**：技术中立 / 无法事先审查 / 已尽通知删除义务 / 仅提供信息存储空间 / 无主观过错 / 损失计算过高。

## 六、与编码、统计、报告的衔接（务必对齐键名）

- **编码**（`05_enriched_cases.json`）：`焦点立场` 的键 = 本文件的焦点名；`抗辩` 的 `理由` = 本文件的抗辩名；**逐字一致**。
- **统计**（`run_analytics.py`）：维度五按焦点名聚合出"立场分布"，并产出"争点出现频次"图；维度二按抗辩名聚合成功率。键名不一致 → 聚合失败、图表残缺。
- **报告**（report-*.md）：第二章每个焦点一节，节内四段展开；综合两表（裁判规则地图 / 控辩指引）逐行对应一个焦点。
- **样本量自适应**：`stats_guard.depth_mode` 为 `qualitative_deep`（案少）时，每个焦点逐案细读、把法院论证链条充分展开；为 `quantitative_lead`（案多）时，量化承载全景、定性只深挖标杆/分歧案。详见 `analysis-dimensions.md`。

## 七、纪律

- 焦点与倾向标签在**检查点 1** 定稿、交律师审核，确认后才进入检索编码；中途若发现遗漏焦点，回到检查点 1 补充并重新确认，不在编码阶段擅自新增键名。
- 无实质分歧的焦点，如实写"本批无明显分歧"，**不得为凑分歧而编造**反向标签。
- 焦点名、标签、抗辩名一经定稿，全流程（编码/脚本/报告）一字不差地沿用。
