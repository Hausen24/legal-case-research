# 输入数据契约（自带数据模式）

没有北大法宝订阅，也能跑本工作流的**分析与产出**链路：你用任意途径（其它案例库、法院公开文书、
人工整理）拿到判决数据，整理成下面的**扁平表**，用 `scripts/general/import_cases.py` 转换成管道
所需的 `03_raw_cases.json`，即可从 SKILL 的"第二步分组/筛查"或"第三步编码"接入，照常产出报告与清单。

> 反幻觉铁律不变：自带数据同样要求每条带**真实案号 + 审理法院 + 可溯源链接**；`verify_report.py`
> 会校验报告引用的案号都在你的数据池中。

## 一、扁平表字段（CSV 或 JSON 均可）

每行/每条 = 一份判决。**必填**列加 ✱，其余可空。

| 列名 | 说明 | 例 |
|---|---|---|
| `案件id` ✱ | 唯一标识，去重用（任意不重复字符串） | `c001` |
| `案件名称` ✱ | 案件全称 | 甲公司与乙公司侵害作品信息网络传播权纠纷 |
| `案号` ✱ | 标准案号 | （2023）京0192民初1234号 |
| `审理法院` ✱ | 法院全称 | 北京互联网法院 |
| `案件等级` | 普通案例 / 公报案例 / 指导案例 / 典型案例（默认普通案例，仅普通案例进分析池） | 普通案例 |
| `案由` | 标准案由 | 侵害作品信息网络传播权纠纷 |
| `审级` | 民事一审 / 民事二审 / 民事再审 | 民事一审 |
| `裁判日期` | YYYY.MM.DD 或 YYYY-MM-DD | 2023.06.18 |
| `地域` | 省/直辖市（留空则由法院名启发式推断） | 北京 |
| `链接` ✱ | 可溯源的判决书链接 | https://… |
| `本院查明` | 事实认定全文（Ascertain） | … |
| `原告诉求` | PlaintiffClaims | … |
| `抗辩意见` | 被告抗辩（DefenseViewpoint） | … |
| `争议焦点` | ControversialFocus | … |
| `本院认为` | 裁判说理全文（Identified） | … |
| `裁判依据` | 所引法条（RefereeBasis） | … |
| `裁判结果` | 判项全文（RefereeResult） | … |

> 正文字段（本院查明/本院认为/裁判依据/裁判结果）越完整，报告的"全文驱动深度"越扎实；
> 缺失处报告会以〔全文补实：…〕显式占位，不会编造。

## 二、转换

```bash
# JSON：一个数组，每个元素是上表字段的对象
python3 scripts/general/import_cases.py <research_dir> --json my_cases.json
# CSV：表头为上表列名（UTF-8）
python3 scripts/general/import_cases.py <research_dir> --csv my_cases.csv
```

转换器会把扁平表映射成与北大法宝 `get_case_list` 一致的记录结构（`Gid/Title/CaseFlag/
CaseGrade/Category/LastInstanceCourt/LastInstanceDate/Ascertain/Identified/RefereeBasis/
RefereeResult/Url/...`），写入 `<research_dir>/03_raw_cases.json`。之后按对应 SKILL 继续：
筛查→编码（`05_enriched_cases.json`）→ `normalize/run_analytics/generate_excel` → 报告。

## 三、最小示例

见 `examples/byod_sample.json`（3 条），可直接：
```bash
python3 scripts/general/import_cases.py /tmp/byod_demo --json examples/byod_sample.json
```
